# SocSpecGen

SocSpecGen contains scripts used to generate custom spectral files for SOCRATES.

The repository is organized as follows: generating model stellar spectral energy distributions in a SOCRATES-readable format, and generating SOCRATES spectral files with selected molecular absorption, UV cross sections, and collision-induced absorption data.

## Installation

```bash
git clone https://github.com/ruizhizhan/SocSpecGen.git
cd SocSpecGen
conda env create -f environment.yml
conda activate SocSpecGen
```

The Python environment is required for running the configuration and script-generation workflow. SOCRATES is also required. The SOCRATES command-line tools, especially `prep_spec` and `Ccorr_k`, must also be available on `PATH` before running generated worker scripts.

## Data Requirements

Large opacity and stellar model data are expected to live in the repository data directories. Before running a case, confirm that the paths selected in `config_data.py` exist locally.

- `stellar_spectra/synphot_trds/` contains the synphot-compatible stellar atmosphere grids used by `prep_stellar_spectra.py`.
- `ExoMol/` contains molecular absorption data, including UV absorption cross sections and line absorption data under the `hdf5/` subdirectories.
- `hitran/` contains HITRAN collision-induced absorption data.
- `spectral_files/` stores generated SOCRATES spectral files.

Opacity data can be obtained from [ExoMol](https://www.exomol.com/) and [HITRAN](https://hitran.org/). The opacity data used for the paper "Reinterpreting the JWST Observations of 55 Cancri e with a Non-Grey General Circulation Model" are available from Zenodo at [https://doi.org/10.5281/zenodo.19388488](https://doi.org/10.5281/zenodo.19388488).

## Usage

The workflow starts by preparing model stellar spectra and then uses those spectra while generating SOCRATES longwave and shortwave spectral files.

1. Use `prep_stellar_spectra.py` to generate model stellar spectra in the SOCRATES stellar spectrum input format. The script supports PHOENIX spectra through `stsynphot` and SPHINX model grids through `util/sphinx_icat.py`. Typical edits are made near the bottom of the file:

   - choose or add a star in `stellar_const`;
   - choose the model grid with `model_type`, such as `phoenix`, `sphinx1`, `sphinx2-condcloud`, `sphinx2-graycloud`, or `sphinx2-graycloud_MLT0.5`;
   - write the output to `stellar_spectra/soc_in/`.

   Run it with:

   ```bash
   conda activate SocSpecGen
   python prep_stellar_spectra.py
   ```

2. Edit `config_data.py` to define the spectral file configuration. This is the main file to edit before generating a new spectral file.

   Important settings include:

   - `MOLEULES_TO_INCLUDE`: selected gas species for the target spectral file;
   - `STAR_NAME`: stellar spectrum name used by the shortwave calculation;
   - `NUM_KTERM`: number of k terms for correlated-k generation;
   - `INCLUDE_CIA` and `INCLUDE_SOLAR_SED`: switches for CIA and stellar SED inclusion;
   - `GAS_LIBRARY`: gas IDs, ExoMol/HDF5 paths, UV cross-section paths, and valid wavenumber ranges;
   - `CIA_LIBRARY`: HITRAN CIA pair definitions, temperature grids, pressure grids, and band limits;
   - `WNEDGES`: spectral band edges, generated by `get_wnedges()`.

3. Run `spectral_file_manager.py` to read `config_data.py`, validate the selected gases and CIA coverage, and generate longwave and shortwave worker scripts plus a Slurm submission script.

   ```bash
   conda activate SocSpecGen
   python spectral_file_manager.py
   ```

   The script creates a case directory under `slurms/`, with a name like:

   ```text
   slurms/sp_b<N>_<TEST_NAME>/
   ```

   Inside that directory it writes:

   - `<case>_lw.py`: longwave worker script;
   - `<case>_sw.py`: shortwave worker script;
   - `<case>.sh`: Slurm submission script.

   The generated workers call SOCRATES tools to create a skeleton spectral file, generate gas and CIA correlated-k data, add UV absorption where configured, assemble the final spectral file, fix invalid NaN records, and validate the final outputs.

   Submit the generated job from the case directory, for example:

   ```bash
   cd slurms/sp_b<N>_<TEST_NAME>
   sbatch <case>.sh
   ```

   Generated spectral files are moved into `spectral_files/sp_b<N>/`.

Also cite the SPHINX-II Zenodo data release used for the model files.

## Repository Layout

- `prep_stellar_spectra.py`: builds model stellar spectrum files for SOCRATES, including PHOENIX and SPHINX model spectra.
- `config_data.py`: stores the gas, CIA, stellar, spectral-band, and run settings used to define a spectral file.
- `spectral_file_manager.py`: generates executable worker scripts and a Slurm script for making SOCRATES spectral files.
- `util/`: helper routines for SPHINX interpolation, wavelength-grid operations, NetCDF conversion, SOCRATES file cleanup, and output validation.
- `ExoMol/`: absorption cross-section data, including UV absorption and line absorption data under `hdf5/`.
- `hitran/`: CIA data used by the spectral-file generation workflow.
- `stellar_spectra/`: stellar model grids and SOCRATES-formatted stellar input spectra.
- `spectral_files/`: generated SOCRATES spectral files.
- `slurms/`: generated per-case worker and submission scripts.

## Reproducibility Notes

For publication use, record the exact `config_data.py` settings, the input opacity data versions, the stellar spectrum model, and the SOCRATES version used to generate each spectral file. Generated Slurm scripts capture the selected gases, band count, and case name, but the scientific provenance also depends on the external ExoMol, HITRAN, PHOENIX, SPHINX, and SOCRATES data/software versions.
